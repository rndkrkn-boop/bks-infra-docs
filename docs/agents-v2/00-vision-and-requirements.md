# Агентная платформа BKS v2 — видение и требования

> Дата: 2026-07-06. Пакет документов `docs/agents-v2/` — целевая архитектура
> агентной системы для «БайкалКварцСамоцветы» (BKS). Основан на аудите живой
> системы от 2026-07-06 (см. [01-current-state-audit.md](01-current-state-audit.md)).

## 1. Бизнес-контекст

Предприятие полного цикла:

```
добыча камня → обработка → производство изделий → продажа изделий
                                └────────────────→ экспорт камня в Китай
```

Сейчас анализируются два отдела: **производство** и **маркетинг**.
Дальше масштаб растёт: подотделы разных производств, сотрудники,
оборудование, экспортное направление.

## 2. Чего мы хотим от системы (формулировка заказчика, разложенная на цели)

| # | Цель | Что это означает технически |
|---|------|------------------------------|
| G1 | Безотказная прод-система | Supervision всех процессов, health-мониторинг с алертами, автоперезапуск, бэкапы, документированное восстановление. Ни один отказ не остаётся незамеченным дольше N минут (сейчас: MemGraphRAG лежал 8 дней незамеченным). |
| G2 | Масштабируемость | Рост по трём осям: количество профилей/доменов, количество параллельных задач, количество sandbox'ов. Архитектурные контракты (карточки, эпизоды, траектории) не зависят от числа воркеров. |
| G3 | Универсальный развивающийся Hermes | Агент накапливает навыки (skills + curator), шаблоны (templates), память (граф + траектории), и его поведение улучшается без ручного переписывания промптов. |
| G4 | Долгие задачи без потери контекста | Карточка kanban = персистентное состояние задачи (журнал прогресса в комментариях); checkpoint'ы; retrieval из графа и банка траекторий вместо раздувания контекста. |
| G5 | Субагенты | Внутризадачная декомпозиция (Hermes subagents) + межзадачная (kanban links parent→child, dispatcher спавнит воркеров). |
| G6 | Активная память | Чтение И запись: каждый воркер обязан оставлять след (эпизод в графе, траекторию в банке, обновление MEMORY.md) — закреплено контрактом, не доброй волей модели. |
| G7 | Исследование оборудования | Поиск недостающих спецификаций, протоколов, методов съёма отчётов; написание скриптов интеграции; помощь в настройке. Отдельный контур безопасности (OT-сеть). |
| G8 | Анализ сотрудников и подотделов | Оргструктура и показатели в графе знаний; строгий PII-контур (уже есть pii-cleaner LoRA). |

## 3. Функциональные требования

### FR-1. Приём и обработка отчётов (уже спроектировано, не работает)
Директор шлёт отчёт (фото/скан/xlsx) в Telegram → director-bot создаёт
карточку → report-processor обрабатывает по накопляемым шаблонам →
результат комментарием в карточку + эпизод в граф.

### FR-2. Исследовательский конвейер
Карточка-запрос на исследование (оборудование, рынок, лучшие практики) →
research/equipment-research воркер → результат: документ + эпизоды в граф +
при необходимости дочерние карточки (гипотезы → эксперименты).

### FR-3. Конвейер оборудования (новый домен, см. [06-domain-equipment-and-org.md](06-domain-equipment-and-org.md))
- FR-3.1: паспортизация — карточка на единицу оборудования → поиск
  спецификаций/мануалов → структурированный паспорт в граф.
- FR-3.2: интеграция — разработка скриптов съёма данных/отчётов
  (Modbus/OPC UA/файловые экспорты/БД вендорского ПО), только read-only,
  деплой через review-required.
- FR-3.3: помощь в настройке — процедуры и чек-листы, исполнение только
  человеком (human-in-the-loop, никакого автоприменения конфигураций).

### FR-4. Оргструктурный анализ (новый домен)
Подотделы, сотрудники, роли, показатели смен → отдельный namespace графа,
PII-очистка на входе, доступ только уполномоченным профилям.

### FR-5. Маркетинговый контур (существует)
market-monitor → analytics → content; дайджесты в mkt-группу.

### FR-6. Самообучение
- Skills создаются/обновляются воркерами (curator включён).
- Шаблоны обработки накапливаются (report-processor templates).
- Банк траекторий пополняется каждым успешным воркером и ретривится
  новыми (Trajectory Memory, см. [04-memory-spec.md](04-memory-spec.md)).

## 4. Нефункциональные требования

| ID | Требование | Целевое значение |
|----|-----------|------------------|
| NFR-1 | Обнаружение отказа любого контура | ≤ 15 мин (алерт в Telegram) |
| NFR-2 | Автовосстановление процессов sandbox | ≤ 2 мин (supervisor restart) |
| NFR-3 | Зависшая карточка (нет heartbeat) | авто-reclaim, повтор ≤ failure-limit, потом block + алерт |
| NFR-4 | Потеря данных при отказе хоста | ≤ 24 ч (ежедневный бэкап kanban.db, эпизодов, Qdrant, ~/.hermes) |
| NFR-5 | Секреты | только GitLab CI/CD Variables → openshell provider; в файлах sandbox не хранятся (уже так) |
| NFR-6 | PII сотрудников | не покидает pii-контур; в общий граф — только после pii-cleaner |
| NFR-7 | OT-сеть (оборудование) | доступ только из выделенного sandbox с explicit allowlist по IP:port, read-only протокольные операции |
| NFR-8 | Долгая задача переживает рестарт воркера/sandbox | состояние восстановимо из карточки + workspace `dir:` |
| NFR-9 | Изменение прод-конфигурации агентов | только через git (profiles/ в bksamotsvety) + sync-profiles.sh |

## 5. Честные ограничения (что НЕ обещаем)

- **Один физический хост (192.168.2.180, GB10)** — это единая точка отказа.
  «Безотказность» в рамках одного хоста = устойчивость к падению процессов
  и быстрое восстановление, а не выживание при отказе железа. HA уровня
  железа требует второй хост (вынесено в Phase 4+ roadmap как опция).
- **Один GPU** — локальный inference (vllm-classifier, whisper, OCR)
  не реплицируется; деградация — fallback на облако через LiteLLM.
- **LLM-надёжность** — модель может ошибаться; критические действия
  (настройка оборудования, публикации, HR-выводы) всегда за человеком
  (review-required паттерн kanban).

## 6. Принципы проектирования

1. **Не изобретать оркестратор** — у Hermes уже есть kanban-ядро
   (atomic claim, dispatcher-daemon, heartbeat, failure-limit, workspaces,
   review-required, tenant isolation). Дефицит был не в коде, а в том,
   что его никто не запускал и не супервизировал.
2. **Карточка — единица работы и единица восстановления.** Всё, что
   должно пережить рестарт, живёт в карточке (описание, комментарии,
   metadata) и workspace `dir:`, а не в голове модели.
3. **Память по слоям с явными контрактами записи** (см. 04): контекст →
   MEMORY.md → граф → траектории. Каждый слой имеет ответ на вопросы
   «кто пишет, кто читает, когда, что именно».
4. **Расширение = новый профиль + новая доска + пресет политики**, а не
   правка ядра. Доменные контуры (equipment, org) добавляются по шаблону.
5. **Безопасность многослойна и по умолчанию запрещает**: OpenShell
   SSRF-guard + explicit allowlist на каждый внутренний сервис; отдельные
   sandbox'ы для контуров с разным уровнем риска (OT-сеть, PII).
6. **Наблюдаемость раньше функциональности**: сначала алерты и health,
   потом новые фичи. Аудит показал, что это главный урок.
