# Roadmap внедрения

Принцип последовательности: **сначала реанимация и наблюдаемость (Phase 0),
потом конвейер (1), потом память (2), потом новые домены (3), потом
самообучение и масштаб (4)**. Каждая фаза заканчивается проверяемыми
критериями (сведены из DoD-секций спецификаций); не начинаем следующую,
пока не закрыт gate предыдущей — иначе строим этажи на песке, как вышло
с kanban-конвейером поверх незапущенного диспетчера.

## Открытые вопросы к заказчику (не блокируют Phase 0–1)

1. Список приоритетного оборудования для паспортизации (3–5 единиц) и
   есть ли у него вообще цифровые интерфейсы (или начинаем с файловых
   экспортов вендорского ПО?).
2. Топология сети завода: достижима ли OT-сеть с хоста 192.168.2.180,
   есть ли сегментация (это определит, где живут скрипты-опросчики E2).
3. Кто кроме директора по производству будет пользователем прод-группы
   (для pairing/whitelist), кто ревьюер review-required карточек по
   оборудованию.
4. Политика по данным сотрудников: что допустимо анализировать, ретеншн,
   кто имеет доступ к org-pii выводам (юридическая сторона — на заказчике).
5. Приоритет китайского экспортного направления: отдельный конвейер
   (перевод, таможенная документация, B2B-переписка) — кандидат на
   Phase 3.5, если бизнес попросит.

## Риски

| Риск | Митигация |
|---|---|
| Входной поток не приживётся у людей (карточки никто не создаёт — как сейчас: 0 задач) | Phase 1.7 обкатка с живыми пользователями; порог входа = «прислать фото в группу», не сложнее |
| MemGraphRAG (исследовательский код XMUDeepLIT) окажется нестабилен под нагрузкой | он уже за API-фасадом; при проблемах заменим бэкенд за тем же контрактом эпизодов/траекторий (контракты — наши, реализация — сменная) |
| Качество mid-tier на реальных отчётах | шаблоны report-processor + eval-golden-карточки (4.3) + эскалация tier'а через `|profile` теги роутера |
| OT-инциденты | контур 06 §3.3: отдельный sandbox, allowlist, read-only, review, rate limit |
| Расползание бюджета на large | метрики §4 + бюджеты LiteLLM (4.4) |
