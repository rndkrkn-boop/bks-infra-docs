# Спецификация оркестрации: kanban, диспетчер, воркеры, субагенты

Опирается на встроенное kanban-ядро Hermes v2026.5.16 (проверено в живом
sandbox): `hermes kanban {init,boards,create,claim,assign,link,comment,
complete,block,unblock,heartbeat,dispatch,daemon,watch,runs,notify-*,gc,...}`,
SQLite `/sandbox/.hermes/kanban.db`, дашборд-плагин `plugins/kanban`.

## 1. Доски (boards)

Одна доска = один поток работ (workstream). Инициализация — `hermes kanban
boards` при деплое (добавить в `setup.sh`/`sync-profiles.sh` идемпотентный шаг).

| Board | Назначение | Основные assignee |
|---|---|---|
| `production` | отчёты, описания процессов, эксперименты | report-processor, structuring, experiment |
| `marketing` | мониторинг, аналитика, контент | market-monitor, analytics, content |
| `research` | исследовательские запросы (лучшие практики, гипотезы) | research |
| `equipment` | паспортизация, интеграция оборудования (Phase 3 — в sandbox bks-equipment своя БД) | equipment-research, equipment-integrator |
| `platform` | задачи самой платформы: инциденты watchdog, техдолг | человек / claude-code |

## 2. Контракт карточки

### 2.1 Обязательные поля при создании

```python
kanban_create(
    board    = "<из таблицы выше>",
    title    = "<Тип: краткая суть, период/объект>",
    body     = "<по доменному шаблону, см. 2.2>",
    status   = "triage",
    assignee = "<профиль-исполнитель>",
    # идемпотентность обязательна для всего, что создаётся автоматикой
    # (cron sweeps, webhooks, пост-хуки) — повторный запуск не плодит дубли
    idempotency_key = "<источник:стабильный-id>",   # напр. tg:msg:12345
)
```

### 2.2 Шаблон тела (обязателен, воркер вправе block'нуть карточку без него)

```
## Вход
<файлы (абсолютные пути в sandbox), ссылки, id эпизодов графа>

## Что сделать
<критерий готовности, измеримый>

## Контекст
<откуда пришло: кто, когда, каким каналом; связанные карточки>

## Ограничения
<дедлайн, «только чтение», «нужен review человека», бюджет обращений к large>
```

### 2.3 Контракт завершения (единый для всех воркеров)

Воркер ОБЯЗАН завершить одним из трёх способов — «тихих» смертей нет:

1. **Успех**: `kanban_comment` (человекочитаемый результат) →
   `kanban_complete(summary=..., metadata={...})`. В metadata минимум:
   `artifacts` (пути/id), `episodes_written` (id эпизодов L3),
   `trajectory_id` (L4), `follow_up_cards` (id созданных потомков).
2. **Нужен человек**: `kanban_comment` (структурированный handoff JSON) →
   `kanban_block(reason="review-required: <что и почему>")`.
3. **Не могу**: `kanban_block(reason="<чего не хватает>")` — с конкретикой
   (какого файла/доступа/данных).

Правило памяти (G6): пункты `episodes_written`/`trajectory_id` — часть
Definition of Done, а не опция. Исключения: чисто механические задачи
(structuring сам и есть запись в память).

### 2.4 Декомпозиция

- **Межзадачная**: `kanban link parent→child`. Потомки исполняются другими
  профилями; родитель complete'ится только после потомков (research создаёт
  карточки-гипотезы → experiment).
- **Внутризадачная**: Hermes subagents — для параллельного сбора
  (несколько источников, несколько единиц оборудования) внутри одного
  воркера. Субагент не трогает kanban; его результат агрегирует родитель.
  Порог выбора: если промежуточный результат должен пережить рестарт или
  исполняться другим профилем — это карточка, иначе субагент.

## 3. Диспетчер

### 3.1 Запуск (под supervision, см. 05)

```sh
hermes kanban daemon --interval 30 --max 2 --failure-limit 2 \
  --pidfile ~/.hermes/kanban-daemon.pid --verbose >> /tmp/kanban-daemon.log 2>&1
```

- `--max 2` — не более 2 спавнов за tick: на одном GB10-хосте больше
  параллельных large-воркеров не прожуёт ни GPU, ни бюджет NVIDIA API.
  Параметр пересматривается по метрикам (05 §4).
- `--failure-limit 2` — после 2 подряд spawn_failed/timed_out/crashed
  карточка блокируется автоматически → watchdog видит blocked → алерт.

### 3.2 Долгие задачи и живучесть

- Воркер обязан слать `heartbeat` (встроено в KANBAN_GUIDANCE);
  отсутствие heartbeat → `reclaim` → задача снова ready.
- Workspace для восстановимых задач — `dir:<персистентный путь>`
  (например `~/.hermes/workspaces/<board>/<task-id>`), НЕ `scratch`:
  промежуточные артефакты переживают рестарт воркера.
- Прогресс фиксируется комментариями с маркером этапа:
  `progress: [3/7] собраны спецификации X,Y; дальше Z` — перезапущенный
  воркер начинает с чтения `kanban context <id>` (комментарии + события)
  и продолжает с последнего этапа, а не с нуля.
- Для очень долгих исследований — чекпоинт правилом «этап ≤ 30 минут»:
  этап заканчивается комментарием-прогрессом; при превышении бюджета
  воркер сам режет остаток на карточки-потомки.

### 3.3 Sweeps (страховочные cron-джобы внутри sandbox)

| Джоба | Расписание | Действие |
|---|---|---|
| `stale-triage-sweep` | каждые 6ч | карточки в triage > 24ч → комментарий + алерт-карточка на board platform |
| `blocked-digest` | ежедневно | сводка blocked (включая review-required) → Telegram прод/mkt-группа по принадлежности |
| `report-processor-sweep` | каждые 12ч | (из README) подобрать зависшие карточки отчётов |
| `kanban-gc` | еженедельно | `hermes kanban gc` — чистка архивных задач и scratch-workspaces |

## 4. Входная плоскость → карточки

### 4.1 Разговорные боты

- director-bot `/report-intake`, `/intake`; mkt-bot `/intake` — уже
  спроектированы. Добавить в каждый intake-skill: `idempotency_key` от
  message_id и **верификацию созданной карточки** (skill уже требует
  проверять id — дополнить: при ошибке kanban_create бот говорит об этом
  явно, а не молчит).
- Подписка на результат: intake-skill делает `kanban notify-subscribe`,
  чтобы бот сам отчитался в группу о завершении.

### 4.2 Планировщик

`hermes cron` per-profile: market-monitor (ежедневный мониторинг),
analytics (weekly-digest), sweeps выше. Все cron-джобы создаются
декларативно из деплой-скрипта (идемпотентно, по `--name`), список джоб —
в git, а не в чьей-то памяти.

### 4.3 Webhook

`hermes webhook` — входы от внешних систем (GitLab CI, вендорское ПО
оборудования, будущий мост между sandbox'ами). Каждая подписка описывается
в profiles/*/config или деплой-скрипте.

## 5. Review-required: человек в контуре

Обязателен для: изменений скриптов интеграции с оборудованием (06),
публикаций контента наружу, любых выводов по конкретным сотрудникам (06),
изменений собственных skills воркеров с влиянием на прод.

Механика (апстримный паттерн kanban-worker):
1. Воркер: comment со структурированным handoff → `block("review-required: ...")`.
2. blocked-digest доставляет в Telegram ответственным.
3. Человек: `hermes kanban unblock <id>` (одобрено — воркер респавнится и
   завершает) либо comment с правками (воркер учитывает при респавне).

## 6. Метрики оркестрации (собирает watchdog, см. 05)

- очередь: карточек в triage/ready, возраст самой старой;
- поток: complete/blocked за сутки по доскам;
- надёжность: доля runs со статусом crashed/timed_out;
- latency: медиана triage→complete по типам задач.

## 7. Definition of Done этой спецификации

- [ ] daemon работает под supervision, переживает рестарт sandbox
- [ ] e2e: отчёт в Telegram → карточка → воркер → комментарий-результат → нотификация в группу (без ручных действий)
- [ ] искусственно убитый воркер: карточка reclaim'ится и завершается со второй попытки
- [ ] карточка с заведомо невыполнимой задачей: 2 попытки → blocked → алерт
- [ ] sweeps и cron-джобы видны в `hermes cron list` после чистого деплоя
