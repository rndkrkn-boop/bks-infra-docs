# Доменные контуры: оборудование завода и оргструктура

Два новых домена с особыми требованиями безопасности. Оба добавляются по
шаблонной процедуре 05 §7.2, но каждому нужен собственный контур риска:
оборудованию — OT-сеть, оргштату — PII.

## 1. Домен «Оборудование» — что заказано

Из постановки: исследования по оборудованию завода — недостающие
спецификации, методы взаимодействия для получения отчётов, формирование
отчётов, настройка оборудования, взаимодействие через скрипты.

Раскладывается на три конвейера с разным уровнем риска:

| Конвейер | Риск | Автономия агента |
|---|---|---|
| E1. Паспортизация (исследование) | низкий (только web + граф) | полная |
| E2. Съём данных/отчётов (интеграция) | средний (read-only доступ в OT-сеть) | код пишет агент, деплой через review-required |
| E3. Настройка оборудования | высокий (запись в контроллеры) | агент готовит процедуру, исполняет ТОЛЬКО человек |

## 2. E1 — Паспортизация оборудования

### 2.1 Профиль `equipment-research` (large, web + граф)

Вход — карточка на единицу оборудования (минимум: наименование, модель,
что известно; фото шильдика — через vision). Выход — **паспорт** и эпизоды
в namespace `equipment`.

### 2.2 Контракт паспорта (структура эпизодов графа + markdown-документ)

```
equipment/<slug>/passport.md   (workspace dir:, versioned)
─ Идентификация: производитель, модель, серийный, год, инвентарный
─ Назначение и место: цех/подотдел, участок процесса (→ связь с prod-графом!)
─ Характеристики: мощность, производительность, расходники
─ Интерфейсы данных: панель/USB/RS-485/Ethernet; протоколы (Modbus RTU/TCP,
  OPC UA, MTConnect, вендорский); ПО вендора и его БД/экспорты
─ Документация: найденные мануалы (PDF в workspace), даташиты, ссылки
─ Пробелы: чего НЕ нашли (→ карточки-потомки: «спросить у механика»,
  «сфотографировать паспортную табличку», «запросить у поставщика»)
─ Обслуживание: регламенты, известные поломки (пополняется из отчётов)
```

Пробелы — первоклассная сущность: карточка-потомок на человека
(assignee=человек через director-bot нотификацию) — так агент сам
организует дозаполнение того, что нельзя нагуглить.

### 2.3 Источники
web_search/web_extract (мануалы, форумы, каталоги поставщиков; включая
китайских производителей — язык не барьер для large-tier), фото шильдиков
и панелей через vision, слова механиков через director-bot intake.

## 3. E2 — Интеграция: скрипты съёма данных

### 3.1 Профиль `equipment-integrator` (mid/large, code_execution + git)

Вход — карточка «наладить съём отчёта с <оборудование>» (пререквизит:
паспорт E1 с заполненными «Интерфейсами данных»). Выход — скрипт в
git-репозитории `bks/equipment-scripts` (GitLab) + документация запуска +
эпизод «как снимать данные с X».

### 3.2 Технические паттерны (по типам интерфейсов)

- Файловые экспорты вендорского ПО (CSV/XLS по расписанию) — самый частый
  и безопасный: скрипт-парсер + доставка в конвейер отчётов (карточка на
  report-processor) — переиспользуем существующий поток FR-1;
- Сетевые протоколы: Modbus TCP (pymodbus, только function codes чтения),
  OPC UA (opcua/asyncua, анонимный или read-only юзер), HTTP API вендора;
- БД вендорского ПО: read-only коннект (отдельный БД-юзер SELECT-only);
- Legacy (RS-485/USB): через шлюз-конвертер или host-скрипт вне sandbox —
  решается по месту, агент готовит код и инструкцию.

### 3.3 Контур безопасности (обязательный, НЕ упрощать)

1. Отдельный sandbox `bks-equipment`; в общий `bks-production` OT-доступ
   не выдаётся никогда.
2. Пресет `ot-network.yaml` в sandbox-templates: explicit allowlist
   `IP:port` per-устройство (как internal-api), добавление устройства =
   PR в sandbox-templates → след в git.
3. Только чтение: протокольные операции записи запрещены на уровне кода
   (linter-правило в CI equipment-scripts: запрет write function codes /
   методов записи OPC UA) и на уровне review.
4. Деплой скрипта в прод (systemd timer на хосте или cron в sandbox) —
   только после review-required (03 §5) человеком.
5. Изоляция от веба: equipment-integrator без web-toolset'а (исследует
   equipment-research, интегратор только кодит) — снижает риск prompt
   injection из интернета в контур с OT-доступом.
6. Премортем: скрипт-опросчик заваливает станок запросами → rate limit в
   шаблоне скрипта (константа интервала, по умолчанию ≥ 5с) + запуск
   новых опросчиков только в нерабочие часы первую неделю.

### 3.4 Поток данных от оборудования

```
станок → скрипт (bks-equipment или host) → сырой файл в spool-каталог
  → карточка (idempotency_key от имени файла) на report-processor
  → обработка по шаблону → эпизоды в prod-граф → analytics
```
Переиспользуется весь существующий конвейер отчётов — оборудование просто
становится ещё одним «директором, присылающим отчёты».

## 4. E3 — Настройка оборудования (human-in-the-loop)

Агент (equipment-research) готовит: пошаговую процедуру с цитатами из
мануала (страница/раздел), чек-лист проверок до/после, план отката.
Карточка завершается `block("review-required: процедура настройки X готова")`.
Человек исполняет у станка; результат (получилось/нет, отклонения) — голосом
или текстом через director-bot → эпизод в equipment-граф → процедура
уточняется. **Никакой записи в контроллеры из sandbox — без исключений.**
Это ограничение фиксируется в SOUL.md обоих equipment-профилей.

## 5. Домен «Оргструктура»: подотделы и сотрудники

### 5.1 Профиль `org-analytics` (large, sandbox `bks-org`)

Задачи: модель подотделов производств (кто что делает, потоки между
участками), показатели смен/бригад из отчётов, узкие места, аномалии
(«участок X стабильно теряет 12% на переделе Y при бригаде Z»).

### 5.2 PII-контур (NFR-6)

1. Namespace `org-pii` — единственное место с именами/персональными
   показателями; ключ ACL только у org-analytics; ретеншн — определить
   с заказчиком (например, показатели старше 2 лет агрегируются).
2. Всё, что уходит из контура (в общий граф `org-public`, в дайджесты,
   в другие профили), проходит **pii-cleaner-lora-v1** (уже задеплоен в
   vllm-classifier роутера) + правило агрегации: наружу — роли и
   агрегаты («бригада 2 смены», «оператор станка X»), не персоналии.
3. Выводы о конкретном сотруднике — всегда review-required карточка
   директору; агент не формулирует кадровых рекомендаций автономно
   (этика + трудовое право + качество данных).
4. Без web-toolset'а: оргданные не соседствуют с внешним контентом.

### 5.3 Источники данных
Отчёты смен (уже идут через FR-1), описания процессов от директора
(/intake), явные карточки-опросники («опиши состав бригады участка X» —
через director-bot человеку).

## 6. Как это ложится на существующий граф

Namespaces `equipment` и `org-*` связываются с `prod` через общие сущности
(станок ↔ участок процесса ↔ подотдел ↔ отчёт). MemGraphRAG-инстанс один,
изоляция — на уровне namespace + ACL ключей (04 §4.2). Кросс-namespace
retrieval (research читает prod+equipment+org-public) — штатный режим для
синтеза: «какие лучшие практики для станка X на участке Y с учётом
показателей бригад» — это и есть целевой сценарий G7+G8.

## 7. Definition of Done (домены)

- [ ] 3 единицы оборудования паспортизованы e2e (карточка → паспорт → эпизоды → карточки-пробелы людям)
- [ ] 1 скрипт съёма данных доведён до прода через review-required; данные текут в конвейер отчётов ≥ недели
- [ ] попытка агента обратиться к OT-хосту не из allowlist блокируется (проверить негативным тестом)
- [ ] орг-модель подотделов производства в графе; первый аналитический отчёт по подотделам принят директором
- [ ] негативный тест PII: имя сотрудника не проходит в org-public/дайджесты
