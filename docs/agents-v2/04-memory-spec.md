# Спецификация памяти: 4 слоя + Trajectory Memory

Цель (G3, G4, G6): агент не теряет контекст на долгих задачах, накапливает
опыт и активно им пользуется. Каждый слой отвечает на вопросы: **кто пишет,
кто читает, когда, что именно, как деградирует при отказе**.

## 1. Обзор слоёв

| Слой | Носитель | Горизонт | Что хранит |
|---|---|---|---|
| L1 — рабочий контекст | context window + `hermes checkpoints` + карточка kanban | одна задача | ход текущей работы |
| L2 — профильная память | `MEMORY.md`/`USER.md`, templates/, skills/ (per-profile, в sandbox) | недели–месяцы | рабочие паттерны профиля, шаблоны, предпочтения людей |
| L3 — граф знаний | MemGraphRAG (SQLite episodes + igraph + Qdrant) | постоянный | доменные факты: процессы, оборудование, рынок, оргструктура |
| L4 — банк траекторий | MemGraphRAG/Qdrant, новая коллекция (MATM-паттерн) | постоянный | КАК решались задачи: последовательности действий с исходом |

Разделение L3/L4 принципиально: L3 — «что мы знаем о мире» (декларативное),
L4 — «как мы делаем работу» (процедурное). Смешивание их в одном retrieval
портит оба (проверено литературой: arXiv:2606.19911, MATM).

## 2. L1 — задача не живёт в голове модели

- Источник истины о состоянии задачи — **карточка** (body + комментарии
  `progress: [k/n] ...` + metadata) и workspace `dir:`. Контекст модели —
  кэш, который можно потерять в любой момент (рестарт, компакция).
- Правило компакции: перед сжатием контекста воркер сбрасывает
  несохранённый прогресс комментарием в карточку (входит в SOUL-правила
  воркеров).
- `hermes checkpoints` — для восстановления сессий разговорных ботов.

## 3. L2 — профильная память

- `memory_enabled: true` у профилей с межсессионными паттернами
  (research, experiment, analytics, report-processor(+templates),
  director-bot(+user_profile)). Оставляем file-based по умолчанию —
  трёхуровневая модель памяти NemoHermes это уже фиксирует.
- **Гигиена**: MEMORY.md растёт бесконтрольно → ежемесячная cron-джоба
  `memory-hygiene` на профиль: перечитать, слить дубли, устаревшее удалить,
  доменные факты (не паттерны!) переселить эпизодом в L3.
- Skills: включить `hermes curator` (background skill maintenance) для
  воркеров, создающих skills. Изменение skill, влияющего на прод-поведение, —
  через review-required карточку (03 §5).
- Внешние memory-провайдеры Hermes (honcho/mem0/…) — НЕ используем:
  дублируют L3 без доменного графа.

## 4. L3 — MemGraphRAG (граф знаний)

### 4.1 Починка (P0, см. аудит §3) и укрепление

1. Перенести memgraphrag + qdrant из K3s в docker-compose на хосте
   (Phase 0.1: K3s избыточен для single-node, кластер декомиссируется);
   данные — bind mounts, порт `:8010`.
2. Восстановить запекание Contriever в образ; CI-gate: smoke-тест
   `TRANSFORMERS_OFFLINE=1` внутри собранного образа.
3. Алерт watchdog на `/health` (см. 05) — «лежит 8 дней» невозможно.
4. Бэкап каталогов данных (episodes SQLite + igraph) и Qdrant — ежедневно.
5. ✅ (2026-07-09) Профили ходят в MemGraphRAG через MCP, не urllib —
   `bksamotsvety/mcp/memgraphrag_mcp.py` (§4.3–4.4). Заодно исправлены три
   бага retrieve-пайплайна, найденные e2e-тестами (AttributeError на
   рестарте, assert на несовпавших ключах сущностей, NaN в
   min_max_normalize на маленьком графе) — retrieve на малых графах раньше
   гарантированно падал.

### 4.2 Namespaces (расширение)

| Namespace | Содержимое | Пишут | Читают |
|---|---|---|---|
| `prod` (есть) | процессы, отчёты, эксперименты | structuring, experiment | research, analytics, content |
| `mkt` (есть) | рынок, цены, площадки | market-monitor | analytics, content |
| `equipment` (новый) | паспорта оборудования, протоколы, схемы съёма данных | equipment-research, equipment-integrator | equipment-*, research |
| `org-pii` (новый) | подотделы, сотрудники, показатели | org-analytics (после pii-контроля) | только org-analytics |
| `org-public` (новый) | обезличенная оргструктура (роли, подотделы, агрегаты) | org-analytics (через pii-cleaner) | research, analytics |

Доступ по namespace реализован (2026-07-09) на уровне MCP-сервера
(`bksamotsvety/mcp/memgraphrag_mcp.py`, env `MEMGRAPHRAG_READ/WRITE_NAMESPACES`
per-profile в `config.yaml`) — это soft-граница: сам API-ключ MemGraphRAG
один на всех и даёт доступ ко всем namespace на сервере, ACL применяется
только внутри MCP-инструментов. Задача 2.4 роадмапа (per-profile API-ключи
с ACL в api.py, настоящая серверная граница) остаётся открытой — актуально,
если понадобится защита от прямого code_execution-обхода MCP. PII-правила — 06 §4.

### 4.3 Контракт эпизода

Пишущие профили вызывают MCP-инструмент `mcp_memgraphrag_store_episode`
(не сырой HTTP — см. §4.1); он один в один транслирует аргументы в
`POST /api/episodes`:

```json
{
  "namespace": "prod",
  "content": "<связный текст факта/наблюдения, самодостаточный без внешнего контекста>",
  "source": "kanban:production:<task-id>",     // обязательная провенансная ссылка
  "episode_id": "<детерминированный, от source+content-hash>",  // защита от дублей (см. аудит 2026-07-05: коллизии episode_id уже чинили)
  "metadata": {"date": "YYYY-MM-DD", "author_profile": "structuring"}
}
```

Правила качества (в SOUL structuring + skill):
- один эпизод = один факт/наблюдение; не сваливать отчёт целиком;
- обязательны даты и единицы измерения; относительные даты → абсолютные;
- противоречие с существующим знанием → отдельный эпизод-корректировка
  со ссылкой на устаревший (граф хранит историю, retrieval ранжирует свежее).

### 4.4 Retrieval-дисциплина

Читающие профили начинают задачу с `mcp_memgraphrag_retrieve` по
формулировке задачи (top-k, k=5..10) и включают найденное в план; ссылки на
использованные эпизоды — в metadata результата (аудитируемость выводов).

## 5. L4 — Trajectory Memory (новое; паттерн MATM, arXiv:2606.19911)

### 5.1 Идея

Каждый успешный воркер — **producer**: публикует сжатую траекторию
решения. Каждый новый воркер — **consumer**: перед работой ретривит top-k
похожих траекторий и использует их как процедурную подсказку. Эффект по
статье: выше success rate и меньше шагов, без дообучения моделей. Это
прямой ответ на G3 («развивается») и G4 (долгие задачи: не переизобретать
процедуру каждый раз).

### 5.2 Хранение (MVP — без нового сервиса)

Новая коллекция Qdrant `trajectories` + таблица в SQLite MemGraphRAG,
эндпоинты в том же api.py:

```
POST /api/trajectories        — публикация
POST /api/trajectories/query  — retrieval (embedding по task_summary)
```

### 5.3 Контракт траектории

```json
{
  "trajectory_id": "traj:<board>:<task-id>:<run>",
  "task_summary": "обработка сменного отчёта цеха резки, xlsx, 12 колонок",  // ключ retrieval
  "profile": "report-processor",
  "outcome": "success | partial | failed",       // failed тоже ценно: анти-паттерны
  "steps": [                                       // СЖАТО, не сырой лог: 5-20 шагов
    "определил тип отчёта по заголовку через vision_analyze",
    "нашёл шаблон templates/shift-cutting-v3 — совпал по колонкам",
    "..."
  ],
  "tools_used": ["vision_analyze", "code_execution", "kanban_comment"],
  "pitfalls": ["в колонке 'выход' единицы кг, не %"],
  "duration_min": 14,
  "source": "kanban:production:<task-id>"
}
```

Сжатие траектории делает сам воркер в конце задачи (один LLM-вызов по
фиксированному промпту из skill `trajectory-publish`) — не хранить сырые
логи, они не ретривятся осмысленно и распухают.

### 5.4 Инъекция consumer'у

Шаг 0 в KANBAN-воркфлоу каждого воркера (через общий skill
`trajectory-recall`, подключается диспетчером так же, как kanban-worker):

```
query(task_summary=<title+body карточки>, profile=<свой>, top_k=3)
→ секция «Похожие прошлые решения» в план работы
→ если top-1 совпадает по типу — следовать её шагам, отклонения фиксировать
```

### 5.5 Эволюция (Phase 4)

- Траектории с высоким reuse → кандидат на **skill** (curator предлагает
  оформить повторяющуюся процедуру в SKILL.md — превращение опыта в код).
- Датасет траекторий → SFT/LoRA-дистилл для mid-tier (training/ pipeline
  роутера уже умеет adapter-SFT) — воркеры дешевеют со временем.
- Метрика полезности: медиана duration_min и turns по типу задачи должна
  падать со временем; A/B — воркер с recall vs без (флаг в config).

## 6. Мягкая деградация памяти (обязательное поведение)

L3/L4 недоступны (как сейчас: 8 дней CrashLoop) — конвейер НЕ встаёт:

- запись: воркер складывает эпизоды/траектории в локальный spool
  `~/.hermes/memory-spool/*.json`; cron-джоба `memory-spool-flush`
  (каждые 30 мин) дозаписывает их в API, когда тот оживёт
  (идемпотентно по episode_id/trajectory_id);
- чтение: retrieval-таймаут 10с → воркер пишет в комментарий
  «память недоступна, работаю без retrieval» и продолжает;
- watchdog алертит независимо (05), но агенты не зависят от алерта.

## 7. Definition of Done

- [ ] MemGraphRAG стабилен ≥ 7 дней, CI-gate на offline-загрузку эмбеддера
- [ ] эпизоды пишутся с детерминированными id, дубли не растут
- [ ] /api/trajectories: publish+query работают, покрыты тестами в CI memgraphrag
- [ ] воркеры publish/recall траектории (видно по metadata карточек)
- [ ] spool-деградация: выключить memgraphrag на час → конвейер работает, после включения spool дозаписан
- [ ] per-profile API-ключи с ACL namespace
